# FIP2 File Format

FIP2 is the plain-text file format used by Frieve Editor.
It stores cards, labels, and links in a structure that is easy to read by humans and to edit safely.

The goal of FIP2 is simple:

| Goal | FIP2 approach |
|------|----------------|
| Human/LLM readability | Clear section markers like `[[CARDS ...]]` and `[Global]` |
| Safe reload after edits | Each card body is delimited by paired marker lines |
| No redundant counts | The number of items is implied by the number of blocks |

The first non-empty line must be exactly:

`FIP2/1`

Optional: you may add one human notice line right after it. The app ignores it.

Encoding:

- Save as **UTF-8 without BOM**.
- When loading, an optional UTF-8 BOM at the very start is allowed.

---
## 2. File structure (top level)

Order of major regions:

1. **Magic line** (exactly `FIP2/1` on the first non-empty line, line 1)
2. **Human notice** (optional but **recommended** for new files): one line of English, e.g.  
   `This is a UTF-8 text file in Frieve Editor's fip2 format.`  
   Parsers **MUST** ignore this line (it is not a structural token).
3. **Blank line** (recommended after the notice)
4. **`[Global]`** — one line per `Key=value` (keys understood by the app)
5. **`[[LABELS type=0]]` … `[[END:LABELS]]`** — card labels
6. **`[[LABELS type=1]]` … `[[END:LABELS]]`** — link labels
7. **`[[LINKS]]`** … `[[END:LINKS]]`
8. **`[[CARDS focus=<id>]]` … `[[END:CARDS]]`**

Whitespace-only lines are allowed between sections.

---
## 3. `[Global]` section

- Starts with a line `[Global]` (trimmed match).
- Following lines until the next `[[...` or `[` section are `Key=value`.
- **Keys** are the settings recognized by the app (e.g. `Version`, `ReadOnly`, `DefaultView`, `Arrange`, `ArrangeMode`, `AutoScroll`, `AutoZoom`, `FullScreen`, `Exit`, `Zoom`, `X`, `Y`, `TargetCard`, limitation keys, `LinkDirection`, `LinkBackward`, `LinkTarget`, `DateLimitationDateType`, `DateLimitationType`, `AutoSave`, `AutoReload`, …).
- FIP2 does not store explicit counts like `Num=`. The app infers cardinality from the blocks.

---
## 4. Labels (`[[LABELS type=n]]`)

- `n` is `0` (card labels) or `1` (link labels).
- Each label is a block:

```
[[LABEL]]
id=<1-based id>
name=<text>
color=<int>
enabled=<0|1>
show=<0|1>
hide=<0|1>
fold=<0|1>
size=<int>
[[END:LABEL]]
```

- Label `id` **MUST** match the 1-based indices used in `labelList` on cards and `labels` on links (same convention as the app).

---
## 5. Links (`[[LINKS]]`)

Each link:

```
[[LINK]]
from=<card id>
to=<card id>
directionVisible=<0|1>
shape=<int>
labels=<comma-separated label ids, may be empty>
name=<text>
[[END:LINK]]
```

Cardinality = number of `[[LINK]]` blocks. No `Num=` line.

---
## 6. Cards (`[[CARDS focus=id]]`)

- `focus` is the focused card id (`TargetCard` / `CardID` equivalent).

Each card:

```
[[CARD id=<card id>]]
title=<text>
visible=<0|1>
shape=<int>
size=<int>
top=<0|1>
fixed=<0|1>
fold=<0|1>
x=<float>
y=<float>
created=<date string>
updated=<date string>
viewed=<date string>
labelList=<comma-separated label ids, may be empty>
drawing=<TDrawing::Encode string, may be empty>
<!--BODY token=<HEX16>-->
... raw body lines (any content except exact end marker line) ...
<!--END_BODY token=<HEX16>-->
[[END:CARD]]
```

### 6.1 Body boundaries

- Start: `<!--BODY token=<TOKEN>-->` (entire line matches after optional whitespace trim for the line only)
- End: `<!--END_BODY token=<TOKEN>-->` (entire line must match exactly)
- **TOKEN** MUST be unique per card in the sense that it is used to locate the correct end marker line for that card.
- The card body is raw text. The app treats it as plain lines and does not parse it as structured markup.
- The app finds the end of the body using the paired end marker line that carries the same `token` value (no extra checksum is involved).

### 6.2 Collision rule

If body text contains a line **identical** to `<!--END_BODY token=...-->`, the parser would terminate early. Writers **SHOULD** detect collisions and regenerate `TOKEN` until safe.

Practical guidance for manual edits: you can usually keep the same token while changing the body content, as long as you do not introduce a line that is exactly equal to the corresponding `<!--END_BODY ...-->` line (and, by convention, avoid adding a line identical to the `<!--BODY ...-->` start marker as well).

---
## 7. Card body editing (the important part)

The card body is the raw text between these two marker lines:

`<!--BODY token=<TOKEN>-->`

and:

`<!--END_BODY token=<TOKEN>-->`

The `token` value is an internal delimiter generated by the app.
In normal use, you should keep the marker lines unchanged and edit only the body lines.

### Collision caution

If the body contains a line that is exactly the same as the `<!--END_BODY token=...-->` line, loading may stop the body early.
If this happens, saving from the app will regenerate a safe token.

---
## 8. Format version

The first non-empty line `FIP2/1` is the format version indicator.
